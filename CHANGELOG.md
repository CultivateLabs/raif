## v1.6.0-pre

- **Behavior Change**: Anthropic JSON responses via the synthetic `json_response` tool (the path used when provider-managed WebSearch suppresses native structured outputs) are now schema-enforced and repair-tolerant. On models that support structured outputs, the tool is sent with `strict: true` so the API constrains the tool input to the schema; constraint keywords that strict mode rejects (`minimum`/`maximum`, `exclusiveMinimum`/`exclusiveMaximum`, `multipleOf`, and length/size bounds) are stripped from the tool's schema, so callers keep their own range validation. Response extraction also normalizes degenerate tool inputs that models occasionally emit: a payload nested under a `json_response` key is unwrapped, a double-encoded JSON string payload is decoded, and a tool input carrying none of the schema's properties is treated as unusable so extraction falls back to text blocks. The same extraction repair now applies to AWS Bedrock's synthetic `json_response` tool (the path used by models without native structured outputs support, e.g. `bedrock_claude_5_sonnet`); Bedrock's Converse API has no strict tool input option, so repair is the only available defense there. The normalization logic lives in a shared `Raif::Concerns::Llms::JsonResponseNormalization` concern included by both adapters. Host apps that pin request cassettes to the previous non-strict Anthropic tool definition will see differences.
- Added Grok 4.5 models: `x_ai_grok_4_5` (xAI adapter, `grok-4.5`) and `open_router_grok_4_5` (OpenRouter, `x-ai/grok-4.5`).
- `bin/smoke_llm_models` now supports xAI models: a new `x_ai` provider selector and `XAI_API_KEY` / `X_AI_API_KEY` wiring, matching the other providers. The `bin/probe_streaming_tool_calls` and `bin/probe_structured_outputs` usage docs now list their existing `x_ai` selector and `XAI_API_KEY` env var.
- **Breaking change**: Removed Claude models that their providers have officially retired. The Anthropic-API keys `anthropic_claude_4_opus`, `anthropic_claude_4_sonnet`, `anthropic_claude_3_7_sonnet`, `anthropic_claude_3_5_sonnet`, `anthropic_claude_3_5_haiku`, and `anthropic_claude_3_opus` map to models listed as **Retired** on [Anthropic's model deprecations page](https://platform.claude.com/docs/en/about-claude/model-deprecations) (for example, Claude 3.5 Sonnet was retired 2025-10-28, Claude 3 Opus 2026-01-05, and Claude 3.5 Haiku and 3.7 Sonnet 2026-02-19), so requests to them now fail. The AWS Bedrock keys `bedrock_claude_4_opus`, `bedrock_claude_3_5_haiku`, and `bedrock_claude_3_opus` have reached end-of-life on Bedrock, which sets its own retirement schedule (see the [Amazon Bedrock model lifecycle](https://docs.aws.amazon.com/bedrock/latest/userguide/model-lifecycle.html)); `bedrock_claude_4_sonnet` remains available (AWS **Legacy** since 2026-04-14, end-of-life 2026-10-14). `bedrock_claude_3_7_sonnet` and `bedrock_claude_3_5_sonnet` are AWS **Legacy** with an end-of-life date of 2026-07-30 and are kept until then, but per the lifecycle table their remaining regions are limited (us-gov-east-1/us-gov-west-1 for 3.7 Sonnet; ap-northeast-1, ap-northeast-3, ap-south-2, and ap-southeast-2 for 3.5 Sonnet v2); other regions, including us-east-1, already return end-of-life errors for them. All three will be removed at their EOL dates. The OpenRouter key `open_router_claude_3_7_sonnet` is also removed. Host apps that reference a removed key (e.g. via `config.default_llm_model_key` or `RAIF_DEFAULT_LLM_MODEL_KEY`) must switch to a current model such as `anthropic_claude_5_sonnet`, `anthropic_claude_4_5_haiku`, or `bedrock_claude_5_sonnet`.
- **Breaking change**: Removed non-Anthropic models that their providers have shut down. `open_ai_o1_mini` and `open_ai_responses_o1_mini`: OpenAI shut down `o1-mini` on 2025-10-27 (see [OpenAI's deprecations page](https://platform.openai.com/docs/deprecations)). The Responses-API key was only ever registered due to a bug (the registry's `delete_if` intended to exclude it ran after key rewriting and never matched), and o1-mini was never actually supported by OpenAI's Responses API, so requests to that key failed regardless. `google_gemini_3_0_pro`: Google shut down `gemini-3-pro-preview` on 2026-03-09 and the alias now redirects to `gemini-3.1-pro-preview`, which is already registered as `google_gemini_3_1_pro`. The OpenRouter keys `open_router_gemini_2_0_flash`, `open_router_gemini_3_pro_preview`, `open_router_grok_4`, and `open_router_grok_4_1_fast` no longer appear in [OpenRouter's model list](https://openrouter.ai/api/v1/models) and return 404s. Also updated `google_gemini_2_5_pro` to target the stable `gemini-2.5-pro` API name instead of the June 2025 preview snapshot `gemini-2.5-pro-preview-06-05`, which Google shut down on 2025-12-02. Note that `gemini-2.5-pro` is itself deprecated (earliest shutdown 2026-10-16, replacement `gemini-3.1-pro-preview`) and Google restricts deprecated models to projects with prior usage, so it can return 404s for projects that have never called it. Host apps that reference a removed key must switch to a current model.
- Added `Raif.config.model_completion_authorizer`, an optional lambda called with `llm:`/`source:` keyword args at the start of `Raif::Llm#chat`, before the `Raif::ModelCompletion` is created or any provider API call is made (and before the `llm_api_requests_enabled` guard, so authorization applies even when API requests are disabled). Raise from it to veto the request — the exception propagates to the caller, including through `Raif::Task.run` (which otherwise rescues `StandardError`): the task is marked `failed` and the original exception is re-raised rather than being reported as an ordinary model failure, so callers can distinguish an intentional veto from a model error. Conversation responses run in a background job with no host caller, so a veto there surfaces as a failed `Raif::ConversationEntry` instead of propagating. Return values are ignored. Useful for host apps enforcing per-account usage limits (e.g. an inference spend budget). Does not apply to embedding generation or batch API submissions. Defaults to `nil` (all requests allowed).
- The admin agent page now shows a per-completion tool-call summary (count plus tool names), so a batch that was rejected over `max_tool_calls_per_iteration`, which leaves no tool invocations, is still visible as the calls the model requested.
- **Behavior Change**: when `max_tool_calls_per_iteration` is set, `Raif::Agents::NativeToolCallingAgent`'s default system prompt now states the per-step limit ("you may make up to N independent tool calls in a single step") instead of an unbounded "several", so the model self-limits rather than over-fanning and getting the whole batch rejected (the prompt is unchanged when the cap is `nil`). The over-cap corrective feedback was reworded to keep inviting batching up to the cap ("You may make up to N independent tool calls per step; make the most important N now and defer the rest to a later step."). Host apps that pin tests/cassettes to the old "Make at most N tool calls per step" text will see differences.
- **Breaking change**: split the model-tool observation API into two orthogonal hooks. `Raif::ModelTool.observation_for_invocation(invocation)` is replaced by `format_result_for_llm(invocation)` (controls the tool-call-result content sent to the model on every subsequent turn). `Raif::ModelTool.triggers_observation_to_model?` is replaced by `triggers_immediate_llm_follow_up?(invocation)` (controls whether Raif creates a synthetic follow-up entry to give the model a same-turn round trip). The old methods are removed; subclasses must rename. The two hooks are independent — tools that want to surface a custom payload to later turns without burning an extra LLM completion can override only `format_result_for_llm`. The conversation-entry helper `triggers_observation_to_model?` is renamed to `triggers_immediate_llm_follow_up?`, and `create_entry_for_observation!` is renamed to `create_immediate_follow_up_entry!`. Admin model-tool-invocation methods `admin_observation` / `admin_observation_error` / `admin_observation_available?` are renamed to `admin_formatted_result` / `admin_formatted_result_error` / `admin_formatted_result_available?`, and the locale keys under `raif.admin.model_tool_invocations.show` (`observation_sent_to_model`, `observation_disclaimer`, `observation_unavailable`) are renamed to `result_sent_to_model`, `formatted_result_disclaimer`, `formatted_result_unavailable`.
- **Behavior Change**: `Raif::Agents::NativeToolCallingAgent` now recovers from rejected and truncated tool calls instead of permanently failing the run. A rejected tool call (unknown tool, invalid arguments, or the wrong tool on a required-tool iteration) is no longer persisted to conversation history — previously it was replayed to the provider on every subsequent request, which providers reject (e.g. OpenAI returns a 400 for a `function_call` input item with no paired `function_call_output`). A completion that was cut off at the max output token limit is discarded entirely, since its tool call arguments may be truncated mid-JSON. In both cases the agent keeps any assistant text and sends the model corrective feedback as a user message, which now includes the source-aware tool arguments schema, a capped excerpt of the arguments received, and JSON schema validator errors. Host apps that render or assert on `Raif::Agent#conversation_history`, or pin tests/cassettes to the old error message text, will see differences on these paths; agent runs that hit them will have different iteration counts and outcomes (typically recovering where they previously failed). [#489](https://github.com/CultivateLabs/raif/pull/489)
- **Behavior Change**: `Raif::Agents::NativeToolCallingAgent` now processes multiple tool calls in a single iteration by default, cutting the number of LLM round trips when the model batches independent calls. Each call in a batch is validated and then invoked in order, with its result interleaved immediately after the call so providers see correctly paired `tool_call`/`tool_call_result` messages on replay. Validation is atomic: if any call in a batch is invalid, none are invoked or persisted and the model receives corrective feedback covering every invalid call. If `agent_final_answer` is returned alongside other tools, the final answer is honored and the sibling calls are dropped without being invoked (so they cannot register results the model never observed). Two new class attributes on `Raif::Agent` control this: `parallel_tool_calls` (default `true`; set to `false` to restore one-tool-call-per-step behavior) and `max_tool_calls_per_iteration` (default `nil`; set to an integer to reject batches larger than the cap). On providers that can disable it (OpenAI completions/responses, OpenRouter, xAI, Anthropic), parallel tool use is turned off at the API level when an iteration forces a specific tool (e.g. the final-answer iteration); ordinary iterations leave the provider default in place (parallel allowed) and the agent processes whatever batch the model returns. Bedrock and Google expose no such switch, so multiple calls there are always handled agent-side. Individual models can opt out by overriding `Raif::Llm#supports_parallel_tool_calls?` (default `true`) — agents AND this into their parallel decision, falling back to one tool call per step when it returns false. The base agent system prompt now tells the model it may make several independent tool calls per step and must call `agent_final_answer` by itself. Host apps that pin tests/cassettes to single-tool-call request bodies or the old "Only one tool call is allowed per step" behavior will see differences. [#490](https://github.com/CultivateLabs/raif/pull/490)- Added truncation detection for model completions. Provider adapters now persist the raw provider finish/stop reason on a new `raif_model_completions.response_finish_reason` column (new migration — run `rails raif:install:migrations` when upgrading; completions will error without it), and `Raif::ModelCompletion#truncated?` reports whether a response was cut off at the max output token limit or, on Anthropic models, because the request exhausted the model's context window (`model_context_window_exceeded`). Streamed responses carry the finish reason through accumulation for Anthropic, Google, and OpenAI Responses, so truncated streams are flagged the same as non-streamed ones. [#489](https://github.com/CultivateLabs/raif/pull/489)
- `Raif::PollModelCompletionBatchJob` no longer notifies the host's error tracker (Airbrake) for transient provider errors. The job already reschedules itself on the exception classes listed in `Raif.config.llm_request_retriable_exceptions` (e.g. `Faraday::ServerError` for upstream 5xx, `Faraday::TimeoutError`, `Faraday::ConnectionFailed`), so per-attempt notifications were noise that didn't correspond to actionable work. Transient errors are still logged at `error` level; non-transient errors are notified and re-raised as before.
- Added optional `subject` (polymorphic) and `config` (jsonb, default `{}`) columns to `Raif::Conversation` for host apps that want to attach a domain object and/or arbitrary configuration to a conversation.
- Added an xAI adapter (`Raif::Llms::XAi`) with support for streaming, developer-managed tools, and native [structured outputs](https://docs.x.ai/developers/model-capabilities/text/structured-outputs) via `response_format: { type: "json_schema" }` on `/v1/chat/completions`. Registered Grok models: `x_ai_grok_4_3`, `x_ai_grok_4_20_reasoning`, and `x_ai_grok_4_20_non_reasoning`. Configure via `Raif.config.x_ai_api_key` / `Raif.config.x_ai_models_enabled` (or `ENV["XAI_API_KEY"]`).
- Added xAI [Batch API](https://docs.x.ai/docs/guides/batch) support.
- Added a tool-call repair loop to `Raif::ConversationEntry`. When a tool call is malformed (unknown tool name, non-hash arguments, schema mismatch, `prepare_tool_arguments` raises), Raif re-prompts the model with synthetic user-role corrective feedback up to `Raif.config.conversation_entry_max_retries` times (default: 2) before marking the entry as failed. Each attempt produces a new `Raif::ModelCompletion` attached to the same entry and is visible in the web admin.
- Added `Raif::Concerns::ToolCallValidation`, a shared validator used by both `Raif::ConversationEntry` and `Raif::Agents::NativeToolCallingAgent`.
- Added `Raif::Conversation#on_entry_finalized(entry:)` hook, called exactly once per entry after the model response has been saved, tool calls validated and invoked, and the entry transitioned to `completed`. This is the correct location for persistent side effects (DB writes, broadcasts, enqueuing jobs). Side effects must not live in `process_model_response_message`, which is invoked on every streaming chunk and every retry attempt.
- Added `Raif.config.streaming_unsupported_model_keys`. When a caller passes a block to `Raif::Llm#chat` for a model key matching the list (entries may be `String`, `Symbol`, or `Regexp`), Raif transparently falls back to the non-streaming path. Default is `[/\Abedrock_gpt_oss_/]`, since Bedrock's Converse streaming endpoint delivers corrupted `tool_use` deltas for `openai.gpt-oss-*`. Set to `[]` to disable.
- Added `bin/probe_streaming_tool_calls` and `bin/probe_bedrock_stream_transport` diagnostic scripts for investigating suspected streaming issues with a given provider+model combination.
- **Breaking Change**: `Raif::Conversation#prompt_model_for_entry_response` now accepts an `extra_messages:` keyword argument and the gem always passes it. Subclasses that override this method must accept the new kwarg (or `**kwargs`) or they will raise `ArgumentError` on upgrade. The documented extension points (`before_prompt_model_for_entry_response`, `system_prompt_intro`, `process_model_response_message`, etc.) are unaffected.
- **Behavior Change**: `Raif::ConversationEntry` can now produce multiple `Raif::ModelCompletion` records per entry (one per repair-loop attempt). The existing `has_one :raif_model_completion` association has been scoped to return the newest attempt. A new `has_many :raif_model_completions` association exposes the full history. Set `Raif.config.conversation_entry_max_retries = 0` to restore the prior one-completion-per-entry behavior.
- **Behavior Change**: Streaming callers for Bedrock gpt-oss models now transparently fall back to non-streaming (see `streaming_unsupported_model_keys` above). Users who want the previous behavior can set `Raif.config.streaming_unsupported_model_keys = []`.
- `Raif::ModelTool.tool_arguments_schema` and `Raif::ModelTool.example_model_invocation` now accept an arity-1 block whose parameter is the calling `source` (typically the invoking agent). Use this to gate fields on per-run caller context without reading global state. `dynamic: true` is implied for arity-1 tool schemas since they must re-evaluate per call. Raif internals (LLM tool formatters, tool-call validation, invocation persistence) thread `source` through via compat helpers (`tool_arguments_schema_for_source`, `prepare_tool_arguments_for_source`, `example_model_invocation_for_source`), so existing subclass overrides whose signatures predate the `source:` keyword continue to work unchanged.

## v1.2.0

- Added streaming support to improve real-time response handling. [#149](https://github.com/CultivateLabs/raif/pull/149)
- Improved JSON parsing by stripping ASCII control characters before parsing. [#153](https://github.com/CultivateLabs/raif/pull/153)
- Added support for OpenAI's Responses API. [#127](https://github.com/CultivateLabs/raif/pull/127)
- Added provider-managed tools system for utilizing tools built into LLM providers:
  - `Raif::ModelTools::ProviderManaged::WebSearch`: Real-time web search
  - `Raif::ModelTools::ProviderManaged::CodeExecution`: Secure code execution
  - `Raif::ModelTools::ProviderManaged::ImageGeneration`: AI image generation
  [#127](https://github.com/CultivateLabs/raif/pull/127)
- Added `response_id` and `response_array` columns to `raif_model_completions` table for enhanced response tracking and provider-managed tools support. [#127](https://github.com/CultivateLabs/raif/pull/127)
- Added a migration checker to warn if the host app is missing any of Raif's migrations. [#129](https://github.com/CultivateLabs/raif/pull/129)
- The AWS Bedrock adapter has been renamed to be more generalized. Added support for Amazon Nova models. [#137](https://github.com/CultivateLabs/raif/pull/137)
- Added support for OpenAI's o-series models. [#155](https://github.com/CultivateLabs/raif/pull/155)
- The `Raif::Conversation` system prompt is now re-built on each conversation entry to ensure it is not stale. [#156](https://github.com/CultivateLabs/raif/pull/156)


## v1.1.0

- Added support for images and files/PDF's. [#106](https://github.com/CultivateLabs/raif/pull/106)
- Added ability to generate embeddings. [#77](https://github.com/CultivateLabs/raif/pull/77)
- Added support for OpenRouter models. [#93](https://github.com/CultivateLabs/raif/pull/93)
- Added a Stats section to the admin interface. [#90](https://github.com/CultivateLabs/raif/pull/90)
- Each model provider's models will be enabled by default if their API key environment variable is present (e.g. OpenAI models will be enabled by default if `ENV["OPENAI_API_KEY"].present?`).
- AWS Bedrock is now disabled by default. This ensures the `aws-sdk-bedrockruntime` gem is not required unless you use AWS Bedrock models. [#94](https://github.com/CultivateLabs/raif/pull/94)
- Added `gpt-4.1`, `gpt-4.1-mini`, and `gpt-4.1-nano` models to the default list of supported LLMs. [#74](https://github.com/CultivateLabs/raif/pull/74)
- Added `claude-4-sonnet` and `claude-4-opus` models to the default list of supported LLMs. [#119](https://github.com/CultivateLabs/raif/pull/119)
- `Raif::ModelTool` subclasses can now define the tool's arguments schema via a `tool_arguments_schema` block. [#96](https://github.com/CultivateLabs/raif/pull/96)
- `Raif::ModelTool` subclasses can now define `tool_description` and `example_model_invocation` via blocks. [#99](https://github.com/CultivateLabs/raif/pull/99)
- `Raif::Task` subclasses can now define a `json_response_schema` block to specify the JSON response schema for the task. [#109](https://github.com/CultivateLabs/raif/pull/109)
- If a `creator` association implements `raif_display_name`, it will be used in the admin interface. [#65](https://github.com/CultivateLabs/raif/pull/65)
- Agent types can now implement `populate_default_model_tools` to add default model tools to the agent. `Raif::Agents::ReActAgent` will provide these via system prompt. [#66](https://github.com/CultivateLabs/raif/pull/66)
- `Raif::ModelTools::AgentFinalAnswer` removed from the default list of model tools for `Raif::Agents::ReActAgent` since answers are provided via `<answer>` tags. [#66](https://github.com/CultivateLabs/raif/pull/66)
- Estimated cost is now displayed in the admin interface for model completions. [#69](https://github.com/CultivateLabs/raif/pull/69)
- `Raif::Conversation` can now be initialized with a `response_format` to specify the format of the response. If you use something other than basic text, you should include instructions in the conversation's system prompt. [#84](https://github.com/CultivateLabs/raif/pull/84)
- `Raif::Conversation` subtypes can now implement `process_model_response_message` to do custom processing of the model response message before it is saved to the database. [#89](https://github.com/CultivateLabs/raif/pull/89)
- `Raif::ModelTool` subclasses can now override `def self.triggers_observation_to_model?` to specify if the tool call's results should be automatically provided back to the model (e.g. a SearchTool might automatically provide the observation back to the model during a `Raif::Conversation`). [#85](https://github.com/CultivateLabs/raif/pull/85)
- `Raif::Task` subclasses can now set a `temperature` for the LLM via `llm_temperature`. [#95](https://github.com/CultivateLabs/raif/pull/95)
- `Raif::Task` subclasses can now set allowed tags and attributes for HTML LLM responses via `llm_response_allowed_tags` and `llm_response_allowed_attributes`. [#103](https://github.com/CultivateLabs/raif/pull/103)
- LLM API requests will now be retried if they fail. This can be configured via `llm_request_max_retries` and `llm_request_retriable_exceptions`. [#112](https://github.com/CultivateLabs/raif/pull/112)
- `Raif.config.task_system_prompt_intro` and `Raif.config.conversation_system_prompt_intro` can now be a lambda that returns a dynamic system prompt. [#116](https://github.com/CultivateLabs/raif/pull/116)
- `ApiError` classes for each model provider have been removed. Instead, API calls will raise Faraday errors. [#117](https://github.com/CultivateLabs/raif/pull/117)
